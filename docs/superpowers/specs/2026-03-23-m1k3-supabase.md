## Doel

De M1K3 Bulk Tracker (`public/M1K3/index.html`) werkt nu volledig met localStorage. Dit is een single-user app voor Mike. Het doel is alle data te migreren naar Supabase zodat desktop en telefoon automatisch gesynchroniseerd zijn via Google OAuth.

## Beslissingen

- **Eén gebruiker** — alleen Mike (Google account). Andere accounts worden geweerd via RLS.
- **Altijd online** — geen offline fallback nodig.
- **Single HTML-bestand** — blijft `public/M1K3/index.html`. Supabase JS SDK via CDN.
- **CRUD** — boodschappen items én oefeningen (push/pull/legs) zijn bewerkbaar via een 5e tab.
- **Presence-as-checked** — `checklist_state` gebruikt geen boolean kolom; een rij die bestaat = aangevinkt (geldt alleen voor boodschappen — fitness state zit in `fitness_day_state.sets_data`).
- **Seeding guard** — `user_settings` tabel met `seeded_at` timestamp voorkomt herseeding na CRUD-verwijderingen.
- **week_year formaat** — altijd 6 cijfers, week zero-padded: `202602` voor week 2 van 2026, `202612` voor week 12.

## Auth flow

De Supabase JS SDK v2 verwerkt de OAuth redirect automatisch via `onAuthStateChange`. Dit is de correcte v2 aanpak:

```js
const { data: { subscription } } = supabase.auth.onAuthStateChange((event, session) => {
  if (session) {
    initApp(session.user)   // seeden + laden + renderen
  } else {
    renderLoginOverlay()
  }
})
```

`getSession()` wordt alleen gebruikt voor de initiële synchrone check bij pageload (voordat het fragment verwerkt is). `onAuthStateChange` vangt daarna de post-redirect sessie op.

**Stappen:**
1. App opent → `supabase.auth.getSession()` + `onAuthStateChange` listener
2. Geen sessie → login-overlay renderen (donker, gecentreerd)
   - Logo + "Inloggen met Google" knop (SVG Google icon)
3. Klik → `supabase.auth.signInWithOAuth({ provider: 'google', options: { redirectTo: 'https://vexxo.be/M1K3/' } })`
4. Supabase redirect terug → `onAuthStateChange` fired met sessie
5. Sessie actief → `initApp(user)` → seeding check → data laden → render
6. Logout-link in user card onderaan sidebar → `supabase.auth.signOut()`

## Database schema

### `user_settings`
| Kolom | Type | Constraints |
|---|---|---|
| id | uuid | PK, default gen_random_uuid() |
| user_id | uuid | FK → auth.users(id) ON DELETE CASCADE, UNIQUE |
| seeded_at | timestamptz | NOT NULL |

Wordt aangemaakt bij eerste login (na seeden). Aanwezigheid = seeding al gedaan.

### `boodschappen_items`
| Kolom | Type | Constraints |
|---|---|---|
| id | uuid | PK, default gen_random_uuid() |
| user_id | uuid | FK → auth.users(id) ON DELETE CASCADE |
| cat | text | NOT NULL |
| label | text | NOT NULL |
| price | numeric(6,2) | |
| sort_order | int | default 0 |

### `exercises`
| Kolom | Type | Constraints |
|---|---|---|
| id | uuid | PK, default gen_random_uuid() |
| user_id | uuid | FK → auth.users(id) ON DELETE CASCADE |
| type | text | NOT NULL — 'push' \| 'pull' \| 'legs' |
| name | text | NOT NULL |
| sets_count | int | |
| reps_label | text | |
| sort_order | int | default 0 |

### `checklist_state`
| Kolom | Type | Constraints |
|---|---|---|
| id | uuid | PK, default gen_random_uuid() |
| user_id | uuid | FK → auth.users(id) ON DELETE CASCADE |
| item_id | uuid | FK → boodschappen_items(id) ON DELETE CASCADE |
| week_year | int | NOT NULL — bijv. `202612` (zero-padded: `202602` voor week 2) |

**UNIQUE:** `(user_id, item_id, week_year)`

**Presence-as-checked pattern (boodschappen only):**
- Item aankruisen = `INSERT INTO checklist_state`
- Item uitvinken = `DELETE WHERE user_id=... AND item_id=... AND week_year=...`
- Week reset = `DELETE WHERE user_id=... AND week_year != huidigWeek`
- Product verwijderen via CRUD → cascade delete wist alle state-rijen automatisch

> Fitness set-completion wordt NIET via dit patroon opgeslagen — dat zit in `fitness_day_state.sets_data`.

### `fitness_day_state`
| Kolom | Type | Constraints |
|---|---|---|
| id | uuid | PK, default gen_random_uuid() |
| user_id | uuid | FK → auth.users(id) ON DELETE CASCADE |
| week_year | int | NOT NULL — zero-padded (zie week_year formaat) |
| day_id | text | NOT NULL — 'maandag'…'zondag' |
| session_done | bool | default false |
| sets_data | jsonb | default '{}' — `{"0": [true,false,…], "1": […]}` (key = exercise sort_order) |

**UNIQUE:** `(user_id, day_id, week_year)`

**Week-reset:** bij elke app-load: `DELETE FROM fitness_day_state WHERE user_id=... AND week_year != huidigWeek`. Oude weken worden niet als historiek bewaard.

**REPLICA IDENTITY:** beide state-tabellen vereisen `ALTER TABLE checklist_state REPLICA IDENTITY FULL` en `ALTER TABLE fitness_day_state REPLICA IDENTITY FULL` zodat Realtime DELETE-events ook de `user_id` bevatten voor de filter.

## Row Level Security (RLS)

Alle 5 tabellen (`user_settings`, `boodschappen_items`, `exercises`, `checklist_state`, `fitness_day_state`) krijgen RLS enabled. Policies voor elke tabel:

```sql
-- SELECT
USING (auth.uid() = user_id)

-- INSERT
WITH CHECK (auth.uid() = user_id)

-- UPDATE
USING (auth.uid() = user_id)

-- DELETE
USING (auth.uid() = user_id)
```

## week_year formaat

```js
function toWeekYear(d) {
  const week = isoWeek(d)        // 1–53
  const year = d.getFullYear()
  return year * 100 + week       // 202602, 202612, 202653
}
```

Altijd een 6-cijferig getal. `202602` < `202612` — sortering en vergelijking werken correct als integer.

## Seeding flow (eerste login)

Guard via `user_settings` tabel — niet via `count = 0` (dat zou re-seeden na CRUD-verwijdering).

```
1. onAuthStateChange → sessie actief → userId bekend
2. SELECT * FROM user_settings WHERE user_id = auth.uid()
3. Rij bestaat? → sla seeding over, ga naar stap 8
4. INSERT alle 13 BOODSCHAPPEN items → boodschappen_items
5. INSERT alle PUSH (7) + PULL (6) + LEGS (6) oefeningen → exercises
6. INSERT INTO user_settings (user_id, seeded_at) VALUES (auth.uid(), now())
7. (stap 6 mislukt door race condition = geen probleem, volgend pageload opnieuw)
8. loadData() → renderApp()
```

Idempotent: tweede pageload → `user_settings` rij bestaat → seeding overgeslagen.

## App load sequence

```
supabase.auth.onAuthStateChange((event, session) => {
  if (!session) {
    renderLoginOverlay()
    return
  }

  seedIfNeeded(session.user.id)
    .then(() => Promise.all([
      loadBoodschappenItems(),   // vult global `BOODSCHAPPEN` array
      loadExercises(),           // vult global `PUSH`, `PULL`, `LEGS` arrays
      loadChecklistState(),      // vult global `checkedItems` Set (item UUIDs)
      loadFitnessState(),        // vult global `fitnessState` Map (dayId → {session_done, sets_data})
    ]))
    .then(() => {
      patchSCHEMA()              // SCHEMA[].exercises ← geladen arrays koppelen
      setupRealtime()
      showTab(localStorage.getItem('active_tab') || 'boodschappen')
    })
})
```

### SCHEMA bridge (patchSCHEMA)

Na `loadExercises()` moeten de geladen arrays terug in `SCHEMA` gezet worden, zodat bestaande `buildFitness()` / `switchDay()` functies blijven werken:

```js
function patchSCHEMA() {
  SCHEMA.forEach(day => {
    if (day.tag === 'push')  day.exercises = PUSH
    if (day.tag === 'pull')  day.exercises = PULL
    if (day.tag === 'legs')  day.exercises = LEGS
  })
}
```

`PUSH`, `PULL`, `LEGS` worden na `loadExercises()` overschreven met de Supabase-data. `SCHEMA` blijft hardcoded qua structuur (welke dag = welk type), maar de inhoud van `exercises` komt uit Supabase.

## Realtime sync

```js
supabase.channel('state-sync')
  .on('postgres_changes', {
    event: '*', schema: 'public', table: 'checklist_state',
    filter: `user_id=eq.${userId}`
  }, () => loadChecklistState().then(updateShopUI))
  .on('postgres_changes', {
    event: '*', schema: 'public', table: 'fitness_day_state',
    filter: `user_id=eq.${userId}`
  }, () => loadFitnessState().then(() => buildFitness()))
  .subscribe()
```

**Vereiste SQL (eenmalig in Supabase SQL editor):**
```sql
ALTER TABLE checklist_state REPLICA IDENTITY FULL;
ALTER TABLE fitness_day_state REPLICA IDENTITY FULL;
```

Zonder dit vuren DELETE-events niet correct met de `user_id` filter.
